用户角色权限设计

2022-01-06 项目设计

在多角色的系统中一般会给不同角色的用户分配不同的权限。权限一般有以下分类：

其中 API 权限是后端权限，其他三个是前端权限，下面从前端角度介绍用户角色权限的设计方案。

权限设计

操作和数据都在页面内，操作权限和数据权限可作为页面权限的子权限（子属性），页面权限包含了所有权限类型。权限既可以做成可维护配置，也可以写死。

  • 可配置

页面绝大多数由菜单触发（少部分由按钮或地址栏输入触发），以菜单为载体不但能配置菜单，还能配置所有权限类型，而且比起“路由”这样一些专业术语，也更能让用户理解。

菜单权限列表是一个树形结构，描述了菜单的从属关系，决定了菜单的生成层级。对于那些不需要生成菜单的页面（由按钮触发或系统自动跳转的页面），可通过 hidden 字段描述该菜单是否显示即可。

实际项目中，一般由菜单管理生成总的菜单权限列表，角色管理时为角色勾选权限，然后在用户管理中为用户赋予角色。

鉴权

  • 路由鉴权

页面权限采用白名单制，不在此名单内的访问和操作都是非法的，实际需求中并不是所有的页面都需要权限，比如 404，登录页，活动页等，这些页面都不需要权限，只需对权限列表内的进行鉴权。

或者将这些不需鉴权页面作为访问白名单加入到我的权限列表进行鉴权，但这需要维护白名单，不推荐：

  • 操作鉴权

跟路由权限一样，并不是所有的操作都需要权限，鉴权列表外的，都不需要鉴权，当然也可根据产品需求按需鉴权。

实际项目中可以，增、删、改、查，通用操作权限，做一些组件或指令封装。
